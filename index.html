Pourquoi le versionning est apparu

Avant Git, la gestion des versions était souvent centralisée, peu efficace, et plusieurs problèmes se posaient:

  • Risque de perte de travail.
  • Difficulté à collaborer et à gérer les conflits.
  • Manque d’outils pour revenir aux versions antérieures facilement.

Le versioning est apparu pour répondre à ces besoins, améliorer la collaboration et sécuriser le code.

En 2005, Linus Torvalds crée Git, un système de gestion de version distribué performant et robuste, suite à la fin de l’utilisation de BitKeeper.

GitLab : histoire et positionnement

GitLab est créé en 2011 par Dmitriy Zaporozhets et Valery Sizov comme une alternative open-source à GitHub.

GitLab se distingue par son intégration complète des outils DevOps, notamment CI/CD natif, et par la possibilité d’auto-hébergement.

Comparatif GitHub / GitLab

Aspect GitHub GitLab
Philosophie Propriétaire sous Microsoft Open source, auto-hébergement possible
Hébergement Cloud uniquement Cloud ou serveurs privés
CI/CD Souvent via outils externes Intégré natif
Workflow GitHub Flow basé sur Pull Request Branches multiples et gestion avancée
DevOps Pas natif Kubernetes et outils DevOps natifs
Communauté Open Source Majoritairement propriétaire Code 100% public

GitLab se démarque par sa flexibilité et son modèle communautaire, tandis que GitHub reste le leader du social coding et de la visibilité open source.

Installation de Git et création de compte GitHub

Installer Git sur Windows

  1. Télécharger Git depuis https://git-scm.com/download/win
  2. Lancer l’installateur, garder les options par défaut
  3. Ouvrir Git Bash et vérifier l’installation : git --version
  4. Configurer votre nom et email pour Git : 
    git config --global user.name "Votre Nom"
git config --global user.email "votre.email@example.com"
Insister sur la configuration ID dès le début pour éviter les erreurs dans les commits.

Créer un compte GitHub

  1. Aller sur https://github.com et cliquer sur « Sign up »
  2. Entrer un email valide, un mot de passe, un nom d’utilisateur
  3. Valider le compte via l’email reçu

Votre compte GitHub sera votre identité dans la communauté open source.

Les bases de Git

Git est un système décentralisé qui permet de créer des snapshots de votre travail.

Insister sur la configuration ID dès le début pour éviter les erreurs dans les commits.

Concepts clés

  • Repository (dépôt) : lieu où est stocké le code et son historique.
  • Commit : snapshot, point dans le temps avec un message description.
  • Branch (branche) : version parallèle pour travailler sur une fonctionnalité.
  • Merge : réunion de deux branches en une seule.
Insister sur la configuration ID dès le début pour éviter les erreurs dans les commits.

Commandes principales

  • git init : créer un dépôt.
  • git add : préparer les fichiers pour le commit.
  • git commit -m "message" : enregistrer les modifications.
  • git status : afficher l’état du dépôt.
  • git log : afficher l’historique des commits.

TP01 : Créer un dépôt Git et le pousser sur GitHub

  1. Créer un nouveau dossier local.
  2. Initialiser un dépôt avec git init.
  3. Créer un fichier README.md avec une description du projet.
  4. Créer un projet web rapide. Ajouter les fichiers avec git add ..
  5. Faire un commit initial avec git commit -m "Initial commit".
  6. Créer un dépôt distant sur GitHub.
  7. Relier le dépôt local au dépôt distant : git remote add origin URL_DU_DEPO.
  8. Envoyer les commits sur GitHub : git push -u origin main. (Bonus: Que signifie ce -u ?)
  9. Rappeler dans ce fichier README.md ce que sont les repositories, les commits, les branches et les merges.
  10. Créer un nouveau commit avec un nom de commit parlant
  11. Envoyer le nouveau commit sur GitHub (Bonus: Quelle commande rapide avez-vous utilisée ?)
  12. Vérifier la présence des fichiers sur GitHub.
  13. Envoyer moi vos liens vers vos dépôts GitHub.

Le branching dans Git

Les branches permettent de travailler sur des fonctionnalités indépendantes sans impacter la branche principale.

Insister sur la configuration ID dès le début pour éviter les erreurs dans les commits.

Commandes utiles

  • git branch : liste les branches.
  • git branch nom_branche : créer une nouvelle branche.
  • git checkout nom_branche : basculer sur une branche.
  • git merge nom_branche : fusionner une branche dans la branche courante.

Les branches facilitent la collaboration et la gestion de versions.

Insister sur les bons usages du branching.

Git Rebase vs Git Merge

Merge crée un commit spécial qui fusionne deux branches, conservant l’historique entier.

Rebase réapplique les commits d’une branche par-dessus une autre, produisant un historique linéaire et plus propre.

Avantages et inconvénients

  • Merge: Historique complet, mais conflits plus visibles.
  • Rebase: Historique plus lisible, mais possible perte de contexte.

Choisir selon le contexte et la politique d’équipe.

TP02 : Branching Git

  1. Créer une nouvelle branche avec git branch feature_x.
  2. Basculer dessus avec git checkout feature_x.
  3. Faire des modifications sur votre projet web, puis committer.
  4. Revenir à la branche principale git checkout main.
  5. Créer un tag de cette version : git tag v1.0. (Bonus: Si vous voulez vous pouvez créer un tag annoté.)
  6. Fusionner la branche : git merge feature_x.
  7. Supprimer la branche : git branch -d feature_x.
  8. Envoyer les changements sur GitHub. Quelle commande utilisez-vous ?
  9. Vérifier sur GitHub la présence du tag et des modifications.
  10. Envoyer moi vos liens vers vos dépôts GitHub.

Cette pratique cible la collaboration et l’intégration continue.

Encourager les étudiants à pratiquer le branching pour bien maîtriser Git.
